# Babylon.js への移植検討メモ

本ドキュメントは、現在の Three.js ベースの WebMMD アプリケーションを Babylon.js へ移植する際の手数（工数）見積もり、推奨するアプローチ、および主要な技術的課題についてまとめたメモです。

---

## 1. 移植の基本方針と見積もり

| アプローチ | 想定工数 / 期間 | 難易度 | 概要 |
| :--- | :--- | :--- | :--- |
| **パターンA: `babylon-mmd` の利用** <br>*(推奨・現実的)* | **約 1〜2 週間程度** | 中 | 既存のBabylon.js用MMDライブラリを採用し、コア（モデル読込・IK・物理）の自作を回避。アプリ層・UI・WebXRの移植に集中する。 |
| **パターンB: コアロジックの自前移植** <br>*(非推奨・スクラッチ)* | **約 1.5〜3 ヶ月以上** | 極めて高 | Three.jsの `MMDLoader` / `MMDPhysics` 等の全コードをBabylon.js向けに移植。パーサーやIKソルバー、物理シミュレーションを自作する。 |

---

## 2. パターンA（既存ライブラリ利用）における主な作業内容

`babylon-mmd`（コミュニティ製のBabylon.js用MMD再生ライブラリ）を採用する場合、移植作業はアプリケーションの基盤と周辺機能の書き換えに絞られます。

1. **描画基盤の移行 (Three.js → Babylon.js)**
   * `Scene`, `Camera`, `Light`, `Engine` 等の初期化処理をBabylon.jsのAPIへ置き換え。
   * トーンマッピング（ACESFilmic）やシャドウ（`PCFSoftShadowMap` 相当）の設定をBabylon.jsのポストプロセス・シャドウジェネレーターへ統合。
2. **ZIPローダーとの連携調整**
   * `zip-loader.js` で解凍された `ArrayBuffer` を, Babylon.jsのローダープラグインへ受け渡すローディング処理の書き換え。
3. **WebXR (VR) 制御の移植**
   * Three.jsの `renderer.xr` から、Babylon.jsの `WebXRDefaultExperience` もしくはカスタムWebXRマネージャーへ移行。
   * コントローラー入力（Lスティックによる移動、X/Yボタンによるカメラ上下移動など）のバインディングを再実装。
   * Quest等モバイルVR向けの最適化（FFR無効化、背景透過処理、DOM Overlayのバグ回避など）の移植・再調整。
4. **アプリ再生ロジック・UIの繋ぎ込み**
   * UI (`ui.js`) からの再生・一時停止・初期ポーズリセット・モデル/モーション切り替え等のイベントハンドリングを、Babylon.jsのアニメーション制御システムとバインド。

---

## 3. 自前移植（パターンB）が非現実的である理由（技術的難所）

1. **MMD特有の仕様の再実装**
   * MMDの再生には、PMX/VMDファイルの解析だけでなく、足などの制御を行う**CCD-IKソルバー**、表情（モーフ）のブレンド処理など、3D数学に基づく複雑な処理が必要です。これを1から実装すると、挙動のバグ（関節がねじれる等）のデバッグに膨大な時間を要します。
2. **物理演算 (Ammo.js) のバインド**
   * MMDの髪や衣服の揺れは、WebAssembly/JS版のAmmo.js（Bullet）と密接に連携しています。Three.jsの `MMDPhysics.js` はAmmo.jsの低レイヤーC++ APIを直接叩いて剛体やジョイントを構築しているため、これをBabylon.js用に移植するのは非常に困難です。
3. **描画（シェーダー）の再現**
   * MMDトゥーンシェーダー（スフィアマップ適用、輪郭線描画など）をBabylon.jsのカスタムマテリアル（Node MaterialやCustomMaterial）で1から組み立て直す必要があります。

---

## 4. 総括
Babylon.jsへの移植を行う場合、開発工数および品質維持の観点から **`babylon-mmd` ライブラリの利用が一択** となります。
このライブラリによりMMD再生コアの移植コストがほぼゼロになるため、開発者は「Babylon.jsの強力なWebXR環境への統合」や「UI/UXのブラッシュアップ」といった本質的なアプリ開発に集中することができます。
